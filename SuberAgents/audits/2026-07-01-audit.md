# Subagent Consultation — 2026-07-01

**Auditor:** SuberAgents SME · **Claude Code:** v2.1.198 · **Scope:** all 14
subagents in `.claude/agents/` + `test-runner`, `.claude/settings.json`, memory.
**Method:** `knowledge-base/best-practices-checklist.md`, plus **two live subagent
tests** (delegation + memory). Findings ranked by impact × confidence.

## Verdict

**Strong roster, one broken feature.** The definitions are unusually
well-crafted — tight least-privilege tools, clean design-vs-build separation, a
consistent `## Hand-offs` convention, explicit output formats, and "verify, don't
rely on memory" instructions throughout. The one real defect was **persistent
memory**, which was silently misfiring. That's now fixed. The remaining items are
refinements to sharpen autonomous delegation and cost.

## Findings

### 🔴 HIGH — Persistent memory was broken for all 6 memory agents *(FIXED today)*
- **What:** `memory: project` combined with a `tools:` allowlist does not engage
  the documented Read/Write/Edit auto-enable (upstream #57507). **Verified live:**
  the `matchmaking` agent wrote `MEMORY.md` to the **repo root**, not
  `.claude/agent-memory/matchmaking/`, and had **no `Edit`** tool.
- **Why it matters:** memory silently landed at the wrong path → never
  auto-injected next session (agents start blind), and all six agents would
  collide on one repo-root file. The README advertised working memory that wasn't.
- **Fix applied:** for `user-research`, `accessibility-ux`, `matchmaking`,
  `trust-safety`, `privacy-compliance`, `security-engineer`: added `Read, Write,
  Edit` explicitly to `tools`; rewrote each memory instruction to name the exact
  path (`.claude/agent-memory/<name>/MEMORY.md`) with read-at-start/update-at-end;
  committed seed `MEMORY.md` files at that path. Full analysis:
  `knowledge-base/memory-and-state.md`.

### 🔴 HIGH — Wrong memory path documented in agents README *(FIXED today)*
- **What:** `.claude/agents/README.md` said each agent keeps "a `MEMORY.md` (under
  `.claude/`)" — implying one shared file; actual is per-agent
  `.claude/agent-memory/<name>/MEMORY.md`.
- **Fix applied:** corrected the path, documented the #57507 caveat and the
  deterministic pattern, linked the KB.

### 🟡 MEDIUM — Descriptions under-use proactivity cues (autonomy) *(recommend)*
- **What:** most descriptions describe capability + boundary but few signal *when
  to reach for the agent unprompted*. `test-runner` ("Use proactively after code
  changes"), `accessibility-ux` ("involve it on all user-facing features"), and
  `trust-safety` ("involve it early") do this well; the other 11 mostly say "Use
  for X."
- **Why it matters:** Claude routes on `description` like a tool description.
  Trigger + proactivity phrasing measurably improves hands-off delegation — the
  core of "autonomous operation."
- **Recommendation (owner approval — wording is subjective):** add a short
  proactivity/trigger clause to each. Examples:
  - `database-architect`: prepend *"Use proactively before any schema, migration,
    or query-performance change."*
  - `security-engineer`: prepend *"Use proactively on auth flows and any change
    touching credentials, PII, or location."*
  - `privacy-compliance`: *"Use proactively whenever a feature collects, stores, or
    shares personal data — before it ships."*

### 🟡 MEDIUM — `test-runner` model could be `haiku` (cost) *(recommend)*
- **What:** `test-runner` runs `sonnet`, but its job is mechanical (detect
  manifest → run command → summarize pass/fail). Docs explicitly recommend routing
  such work to Haiku.
- **Recommendation:** set `model: haiku`. Keep `sonnet` only if you want it to
  interpret complex failures itself (it currently defers real diagnosis to
  `qa-accessibility-test`, so haiku fits). Low risk; owner's call.

### 🟢 LOW — Sensitive memory is committed (`project` scope) *(decision to make)*
- `security-engineer`, `trust-safety`, `privacy-compliance` memory (threat models,
  known-issue register) is `project` scope → **committed to git**. Fine for a
  private repo; reconsider `local` if this repo could ever be public.
- **Tradeoff:** `local`/`user` memory does **not** survive this ephemeral cloud
  container — only `project` (committed) persists across sessions. So `project` is
  the pragmatic default here; just be deliberate about it.

### 🟢 LOW — Optional tuning
- Consider `effort: high` on the six opus judgment agents for deeper reasoning on
  high-stakes calls (small cost, better rigor).
- `maxTurns: 25` is set on opus agents but not sonnet ones — harmless; add if you
  want a hard ceiling on implementation loops.
- Agent Teams flag is enabled but idle (correct) — reach for it only for the
  frontend/backend/database "align on a contract live" case.

## What's already good (keep doing)
- Least-privilege tool scoping (design agents read-only; implementers get
  Edit/Write/Bash).
- Clear ownership boundaries + `## Hand-offs` routing (works with the
  "orchestrator conducts, agents can't call peers" model).
- Explicit output formats and terse-output instructions for agent-consumed results.
- "Verify current standards, don't rely on memory" on every time-sensitive agent.
